
#  Meeting Notes — 2026-03-20
**Participants:** Anna, Sergey, Artem  
**Format:** Team discussion / sync

## Agenda
- Discuss Firebase data structure
- Define learning content format
- Align on what belongs to frontend vs backend/data
- Decide how to keep the implementation simple

## Discussion

This was one of the most important meetings in terms of architecture. We needed to agree on how the learning content should be structured, because by this point both the UI and Firebase work were progressing, and without a clear model we risked building incompatible pieces.

Anna approached the problem mainly from the user flow side: topics should have an order, difficulty should be visible, and the learning sequence should be predictable. Sergey was more focused on keeping the backend structure manageable and separating user data from theory and test content. Artem’s role in the discussion was more around how the data structure would affect what could be displayed and styled in the interface.

After discussing different options, we agreed not to overengineer the content model. The chosen structure was intentionally simple:

- `id`
- `order`
- `difficulty`
- `title`
- `content`
- `imageUrl`

This gave us enough control for the frontend without forcing a heavy or overly normalized structure too early. We also agreed that even if all theory was stored in one `content` field, it could still remain structured by writing it in Markdown sections.

This meeting helped us separate two concerns clearly:  
1. how content is stored  
2. how the application uses that content in routes and UI

## Decisions
- Keep learning content structure minimal but scalable
- Use `order` to control sequence
- Use `difficulty` for filtering and display
- Store theory in **Markdown-friendly content**
- Avoid premature complexity
