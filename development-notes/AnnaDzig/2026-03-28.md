## Reflecting thoughts about Routing Approach Decision

We considered a few different approaches for handling routing and access control in the application.

### Option 1 — Page-level authentication checks (not chosen)

One option was to keep a flat routing structure and handle authentication directly inside each page component.

In this approach, every page such as `Dashboard`, `UserPage`, or `LessonPage` would check the auth state on its own and decide whether to render content or redirect the user.

Example:

    if (!user) {
      return <Navigate to="/login" />;
    }

### Why we did not choose this approach

Although this solution looks simple at first, it becomes problematic as the application grows.

- **Code duplication**  
  The same authentication check would need to be repeated in many page components.

- **Harder maintenance**  
  If the auth logic changes, we would have to update it in multiple places.

- **Higher risk of mistakes**  
  It becomes easy to forget adding the check to a new protected page.

- **Mixed responsibilities**  
  Each page would contain both page UI logic and access-control logic, which makes components less clean.

### Why we chose layout-based routing instead

Instead of handling authentication inside every page, we moved access control into dedicated layouts:

- `RootLayout` for shared application structure
- `AuthGateLayout` for guest-only pages
- `ProtectedLayout` for authenticated pages

This gives us a cleaner and more scalable architecture because:

- access logic is centralized
- pages stay focused on UI
- duplication is reduced
- the route tree is easier to read and extend

### Summary

We did not choose page-level authentication checks because that approach does not scale well and leads to repeated logic across the application.

We chose layout-based routing because it provides a clearer separation of concerns, keeps components cleaner, and makes the routing structure easier to maintain as the project grows.
