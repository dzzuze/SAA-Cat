# Date: 2026-03-20

- **What was done:**

Today we had a really important discussion with the team about how we should structure our data in Firebase, especially for the learning content. At first, we had slightly different perspectives. I was thinking more about how the app will work from the user’s point of view — how topics are displayed, how learning flows, and how we control the order. My teammate was more focused on keeping the data simple and separating articles from tests.

After talking it through, we found a good balance. We decided that for now, especially considering that I have the CoreJS interview on Monday, it makes sense not to overcomplicate things and move forward with a clean but simple structure for the advanced learning content.

We agreed to use the following structure:

- id
- order
- difficulty
- title
- content
- imageUrl

This structure feels like a good compromise. It is simple enough to quickly prepare learning materials, but still gives us control over how topics are displayed and organized. The `order` field is especially important, because it allows us to define a clear learning sequence instead of relying on random or alphabetical ordering.

I also realized that even though everything is stored inside the `content` field for now, we can still keep it structured by writing it in sections (definition, explanation, examples, etc.), so it remains readable and usable in the UI.

- **Decisions / Notes:**

We decided to keep the structure minimal for now to avoid overengineering, especially with the upcoming interview. At the same time, we made sure to include the key fields that will matter later, like `order` and `difficulty`.

A key takeaway for me was understanding the difference between:

- content structure (articles)
- application structure (topics, order, logic)

We are starting simple, but in a way that will not block future improvements.

- **Problems / Questions:**

The main challenge was aligning our understanding of what exactly we are building at this stage. It was not really a technical problem, but more about architecture thinking and making sure we don’t mix different concerns too early.

- **Plans / Next steps:**

- Prepare JSON file with advanced topics using the agreed structure
- Fill it with CoreJS#2 interview topics
- Use it in the UI to render topics in the correct order
- After the interview, revisit and possibly extend the structure
